# v4.0.0 Release Notes

## Summary
This release introduces a major architectural shift to an event-driven system for timer management, replacing the previous callback-based approach. This change enhances flexibility and extensibility, allowing users to subscribe to various timer lifecycle events. The logging dependency `@logtape/logtape` has also been removed. This is a major version bump due to several breaking changes.

## Added
- **Event-Driven Architecture:** Introduced a new `EventEmitter` class and integrated it into `TimersManager`, enabling an event-driven approach for handling timer lifecycle events.
- **New Events:** `TimersManager` now emits `started`, `stopped`, `expired`, `updated`, `interval`, and `errored` events, providing granular control and observation of timer states.
- **`on`, `once`, `off`, `offAll` methods:** Added to `TimersManager` for event subscription and management.

## Changed
- **`TimersManager` Base Class:** `TimersManager` now extends the new `EventEmitter` class.
- **`checkTimers` replaced by `checkStart`:** The method for initiating timer monitoring is now `checkStart()`, which no longer accepts a callback. Instead, users should subscribe to the `expired` event.
- **Updated `createTimer`, `removeTimer`, `adjustRemainingTime`:** These methods now emit `started`, `stopped`, and `updated` events respectively.
- **`changeExtra` method (in `JSONLTimersManager`):** Now emits an `updated` event when a timer's extra data is modified.

## Fixed
- No specific bug fixes identified in this version.

## Removed
- **`@logtape/logtape` Dependency:** The optional logging dependency and its integration (`src/Log.ts`) have been removed, simplifying the package and reducing its footprint.
- **Callback parameter from `checkTimers`:** The callback function is no longer part of the timer checking mechanism, replaced by the `expired` event.

## Breaking Changes
- **Migration from Callbacks to Events:** The most significant breaking change is the shift from `checkTimers` with a callback to `checkStart` combined with event listeners (`manager.on('expired', ...)`). Existing code using `checkTimers` will need to be updated.
- **Removal of `@logtape/logtape` Integration:** If you were relying on the internal logging provided by `@logtape/logtape`, you will need to implement your own logging solution externally.

## Migration Notes
- **Updating Timer Monitoring Logic:**
    - Replace calls to `manager.checkTimers(callback, interval)` with `manager.checkStart(interval)` to initiate the monitoring loop.
    - Implement an event listener for expired timers: `manager.on('expired', (timer) => { /* handle expired timer */ });`
    - Remember to use `clearInterval()` on the returned value from `checkStart()` to stop monitoring.
- **Handling `updated` events:** If you need to react to changes in timers (e.g., after `adjustRemainingTime` or `changeExtra`), subscribe to the `updated` event: `manager.on('updated', ({ old, new: updatedTimer }) => { /* handle updated timer */ });`
- **Logging:** If you require logging, integrate a logging library of your choice directly into your application.

## Notes
- This release lays the groundwork for more robust and flexible timer management, aligning with modern asynchronous programming patterns.
