## Summary
This release introduces a significant refactoring of the timer management system, making it more flexible and extensible. The core change involves the introduction of a generic 'extra' field for timers, especially impacting JSONL storage. This is a major version bump due to several breaking changes.

## Added
- Introduction of a generic 'Extra' type parameter for `Timer`, `CreateTimerOptions`, `TimersManager`, and `TimersStore` classes, allowing for flexible custom data storage with timers.
- New `changeExtra` method in `JSONLTimersManager` to modify the generic `extra` field of a timer.

## Changed
- The `TimersManager` and `TimersStore` classes are now generic (`<T extends StorageType, Extra extends object>`).
- The constructor documentation for `TimersManager` has been updated.
- The documentation for `createTimer` has been updated.
- `removeTimer`, `checkTimers`, `adjustRemainingTime` JSDoc comments have been updated with type information.

## Fixed
- No specific bug fixes identified in this version based on commit history.

## Removed
- `changeTitle` and `changeDescription` methods from `JSONLTimersManager`. These functionalities are now covered by the more general `changeExtra` method.

## Breaking Changes
- **Generic Timer Type:** The `Timer` type now includes an `extra: Extra` field (`Timer<T extends StorageType, Extra extends object>`). This replaces the optional `title` and `description` fields for `JSONL` timers.
- **Generic `CreateTimerOptions`:** `CreateTimerOptions` for `JSONL` timers now requires an `extra: Extra` object and no longer accepts just a number as a direct duration.
- **Generic `TimersManager` and `TimersStore`:** Both `TimersManager` and `TimersStore` classes are now generic (`<T extends StorageType, Extra extends object>`). This impacts all implementations and extensions of these classes.
- **`JSONLTimersManager` Method Changes:** The `changeTitle` and `changeDescription` methods have been removed. Use the new `changeExtra` method instead.
- **Stricter `JSONL` Timer Syntax:** `JSONLTimersStore` now enforces a stricter syntax for `JSONL` timer files, expecting exactly 4 keys (`id`, `start`, `stop`, `extra`) and requiring the `extra` field to be an object. Existing `JSONL` timer files that do not conform to this new structure (e.g., still having `title` or `description` but no `extra` field) will likely cause parsing errors.

## Migration Notes
- **For `JSONL` Timer Users:** If you were using the `title` or `description` fields with `JSONL` timers, you will need to migrate your data and update your code. The data should now be stored within the `extra` object. For example, `{ id: "...", start: ..., stop: ..., title: "My Timer" }` should become `{ id: "...", start: ..., stop: ..., extra: { title: "My Timer" } }`.
- **Creating `JSONL` Timers:** When creating `JSONL` timers using `createTimer`, you must now provide an `extra` object, even if it's empty (e.g., `{ length: 5000, extra: {} }`). Passing only a number for the duration is no longer supported for `JSONL` timers.
- **Custom `TimersManager` or `TimersStore` Implementations:** If you have custom implementations extending `TimersManager` or `TimersStore`, you will need to update them to include the `Extra` generic type parameter and adapt to the changes in method signatures.
- **Direct `JSONL` Timer File Manipulation:** If you directly manipulate `JSONL` timer files, ensure they conform to the new 4-key structure with the `extra` field as an object.

## Notes
- This release aims to provide greater flexibility for extending timer data in the future.
